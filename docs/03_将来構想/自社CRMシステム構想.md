# 自社CRMシステム構想（将来構想）

> ⚠️ **構想段階**（2026/6/11 言語化）。クライアントワークを支える社内基盤としての構想。

---

## 狙い（一言）

**ココナラが肩代わりしている機能を自社で代替するシステムを作り、そこに本リポジトリで組み立てた
サービスルール（共通基盤＋サービス別＋案件台帳）を載せる。**
→ 手数料ゼロ・高客単価・自社データ蓄積を両立した、**属人化しない「完璧なクライアント対応の仕組み」**。

これは `01_共通基盤/04_チャネル戦略.md` の「自社サービスは高客単価でやれるが、集客・受発注の
仕組みを自前で用意する必要がある」という課題に対する打ち手にあたる。

## ココナラが今担っている機能 → 自社CRMで代替する範囲

| ココナラの機能 | 自社CRMでの代替 | 備考 |
| --- | --- | --- |
| 見積もり提案 | 見積もり作成（カテゴリ×Lv×工数係数を内蔵） | 構造化見積もりをロジック化 |
| メッセージ／トークルーム | 顧客とのやり取り・履歴管理 | |
| 受発注・進行管理 | 案件ステータス（S0〜S6）管理 | 共通業務フローをそのまま実装 |
| 検収・納品確認 | 納品・検収フロー | |
| 評価・実績カウント | 実績/評価の蓄積 | |
| （集客） | **プラットフォーム依存。自社では別途集客が必要** | ここが自社化の最大の課題 |

> 集客機能だけはプラットフォームの本質的価値であり、自社CRMでは置き換えにくい。
> 集客は別チャネル（自社サイト＝ピクセルスタジオ／逆輸入戦略等）で補う前提。

## 本リポジトリのルールとの接続

CRMの「ロジック」として、本リポで定義済みのルールをそのまま組み込む想定：

- **共通業務フロー S0〜S6**（→ `01_共通基盤/01_共通業務フロー.md`）＝ 案件ステータス遷移
- **構造化見積もり**（→ `01_共通基盤/02_見積もり設計フレーム.md`）＝ 見積もり自動組み立て
- **調整費ルール**（→ `01_共通基盤/03_調整費・トラブル対応.md`）＝ 停滞検知・課金
- **サービス＝工程モジュール／クロスセル・アップセル**（→ `00_設計思想`）＝ メニュー構成ロジック
- **案件ログ（受注・失注）**（→ `02_サービス/イラスト制作_ヨンマルヨン/08_…`）＝ CRMの顧客・案件データ

## 期待効果

- **手数料ゼロ・高客単価**（自社チャネル化）。
- **データが自社に貯まる**（CRMとして顧客・案件・受注/失注を一元管理）。
- **ルールをシステムに埋め込む**ことで、対応品質が担当者に依存しない（属人化排除の徹底）。
