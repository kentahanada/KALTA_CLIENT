# 00. 設計思想（共通の根っこ）

カルタのクライアントワーク（Web制作・イラスト/デジタルコンテンツ制作）に共通する、
**根っこの設計思想**をまとめる。各サービスの詳細はこの思想の上に枝分かれする。

---

## 0. 一段上の宣言：思想の核（2026/6/11 言語化）

> **抽象的・感覚的なものを、誰でも再現できる「構造」に翻訳し、属人性を排しながら価値を積み上げる。**
> さらに、**サービスはすべて「工程モジュール」に分解されており、成果物はモジュールの連結で組み上がる。**
> Webもデジタルコンテンツも、この同じエンジンの適用先が違うだけで、幹は同一。

### 0-1. 共通の動作とサービス別の現れ方（幹と枝の対応表）

| 共通の動作（幹） | Web制作での現れ方 | デジタルコンテンツ（イラスト）での現れ方 |
| --- | --- | --- |
| 抽象 → 構造へ翻訳 | 目的・世界観を情報設計(IA)・UIに落とす | 「〇〇風」をスタイルコード（7〜9軸）に落とす |
| 難度を共通言語化 | 思考難度Lv1〜5／技術・運用難度 | 同じ思考難度モデル＋作り込みLv |
| 見積もりを構造化 | カテゴリ×Lv×工数係数（サイト/ページ/セクション3軸） | カテゴリ×Lv×工数係数（＋特急・大判加算） |
| 上流で認識ズレを潰す | ワイヤー/要件定義で合意 | ラフ段階でスタイル認識合わせ（清書後修正NG） |
| 属人化を排す | 共通業務フローS0〜S6・SOP化 | 同じS0〜S6・発注シート・SOP |
| 停滞はコスト化 | 調整費 1営業日¥2,000 | 同じ調整費ルール |

### 0-2. サービス＝工程モジュールの集合（クロスセル × アップセル設計）

サービスは「完成品の一枚メニュー」ではなく、**制作フローを工程単位に分解したモジュール群**として
設計されている。クライアントの欲しい成果物は、**モジュールをフローに沿って連結**して組み上げる。

- **横の連結＝クロスセル**：サービスをまたいでスキルを接続する。
  - 例（Live2Dが欲しい場合）：
    `イラストサービス「キャラクターデザイン」＋「イラスト制作」＋「パーツ分け（オプション）」`
    → `Live2Dサービス「モデリング」` と連結して完成。
  - Webも同構造：`戦略/設計 → デザイン → コーディング → CMS構築 → SEO運用/保守` を
    必要な区間だけ連結して受注する（環境構築・相談などの前後モジュールも単体出品）。
- **縦の深さ＝アップセル**：各モジュール内のレベル（Lv・プラン）で深掘りを売る。
  - 例：モデリングなら可動域のグレード（ライト→ベーシック→アドバンス）。
  - 例：Webならブランディング・戦略コンサル（Aプラン→Bプラン）。

```
        ── 横：フロー連結（クロスセル）──▶
  [キャラデザ] → [イラスト制作] → [パーツ分け] → [モデリング]
      │Lv1-3        │Lv1-3          │+50%         │Light/Basic/Advance
      ▼縦：深さ（アップセル）
```

この設計のメリット：
- 工程単位で**切り売り・部分受注**ができる（「モデリングのみ」「デザイン〜パーツ分けまで」等）。
- 1案件が**複数モジュールの受注機会**になる（クロスセルの起点）。
- 見積もりが「連結したモジュールの積み上げ」になるため、**構造化見積もり（§2）とそのまま噛み合う**。
- 新サービス（動画・ドット絵等）も「モジュールを足す」だけで既存フローに接続できる。

> 実例：ココナラ実出品（`02_サービス/イラスト制作_ヨンマルヨン/07_…`）の
> ミニキャラLive2D「デザイン〜モデリング¥45,000／デザイン〜パーツ分け¥25,000／モデリングのみ¥22,000」は
> 工程モジュールの切り売りそのもの。Web側の「相談¥3,000→コーディング¥15,000→HP制作¥350,000→SEO運用」も
> 同じ連結構造（`02_サービス/web制作/01_…`）。

## 1. 属人化を排し、誰がやっても同じ品質・進行に揃える

- 全サービスで**共通の業務フロー（S0〜S6）**を必ず踏む（→ `01_共通業務フロー.md`）。
- 抽象的・感覚的な指示を排し、**コード化・フォーマット化・SOP化**する。
  - 例：イラストの「〇〇風」を禁止し、スタイルを軸でコード化する（スタイル認識合わせSOP）。
  - 例：Web/イラストの見積もりを「カテゴリ × Lv × 工数係数」で構造化する。
- ドキュメントは「定義 → 手順 → 判定」「条件 → 分岐 → 行動」などの**型（フレーム）**に沿わせる。

> Notion運用ではドキュメントに `DSP（定義→手順→判定）` / `TAO（トリガー→アクション→出力）` /
> `CBA（条件→分岐→行動）` / `CL（チェックリスト）` のフレームを付与して管理している。

## 2. 見積もりは「難度 × 工数」で構造的に積み上げる

- 価格は感覚ではなく、**カテゴリごとにレベル(Lv)を定義 → 工数係数 → 価格目安**で算出する。
- 「思考難度」をマクロにレベル分けして、案件の重さを共通言語で語れるようにする
  （→ `02_見積もり設計フレーム.md`）。
- 標準枠を超える非定型案件（巨大サイズ・短納期など）は、**加算オプション**として明文化する
  （特急料金・大判サイズ制作費など。イラスト側 `05_価格ルール` 参照）。

## 3. 認識ズレを「上流」で潰す

- 制作に入る前に、必ず**要件定義・ヒアリング・スタイル/方向性の合意**を取る。
- ラフ／ワイヤー段階で必ずクライアント確認を挟み、清書・本実装後の手戻りを防ぐ。
- 「やること／やらないこと（作業範囲外）」を見積もり時点で明示する。

## 4. クライアント都合の停滞にはコストを定義する

- クライアント側の確認・素材提供の遅延で進行が止まった場合は、**調整費**を発生させる
  （→ `03_調整費・トラブル対応.md`）。
- 「制作側の不備による遅延」とは明確に区別する。

## 5. プラットフォーム特性を前提に運用する（ココナラ）

- ココナラには**アフターサポートの仕組みがない** → 納品後対応は別契約とする。
- 見積もり提案は必ず**サービスと紐付け**て作成する（実績カウントに必要）。
- 検収・評価の取得までを「納品完了」の条件に含める。

## 6. 共通思想・サービス固有の切り分け

| レイヤー | 内容 | 置き場所 |
| --- | --- | --- |
| 共通（根っこ） | 業務フロー、見積もり設計の思想、調整費、トラブル原則 | `01_共通基盤/` |

> サービス固有の差分だけを各サービス配下で育てられる。
