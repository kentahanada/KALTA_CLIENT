## 診断：なぜ「甘い」のか（優先度順）

### 🔴 D1. モジュールの「粒度・境界」がバラバラ（根本原因）
- コーディング＝工程／web制作＝パッケージ／SEO＝サブスク／サポート＝相談、と**抽象度が4種混在**。
  足し合わせ方（見積もりの組み立て）が定義できない。
- **02_02 と 02_01 の二重定義＋循環参照**：web制作の中に「構築費はコーディング参照」。
  デザインと実装、設計と構築の責任分界が曖昧。
- **範囲外定義が実態と矛盾**：02_02は「サーバー・ドメイン準備／運用代行は範囲外」だが、
  実出品には**サーバー代行¥5,000・SEO運用¥200k**がある（→ `01_ココナラ現行出品ラインナップ（花田ケンタ）.md`）。

### 🔴 D2.「Lv」が複数の意味で使われ、共通言語になっていない
- 複雑さ（コーディング：ページLv×レスポンシブLv）／作り込み（web制作：×1.0-2.0）／
  関与深度（SEO：Lv1-5）／思考難度（マクロモデル：Lv1-5）。
- イラストの「スタイルコード」に相当する**明確な軸定義が無い**ため、Lvが効いていない。

### 🟠 D3. 見積もりの「組み立てルール」が無い（コーディング以外）
- 02_02は料金レイヤーが4つ（戦略コンサル／サイト設計／ページデザイン／コンテンツ実装）あるが、
  **総額をどう積み上げるかの式が無い**。サンプル（KADODE ¥420,000）はあるが再現不能。
- コーディングだけ式が明確（ページLv×レスポンシブLv／px基準）＝ここが全体の手本になる。

### 🟠 D4. A/Bプラン（見やすく伝わる／売れる）が宙に浮いている
- 概念は良いが、モジュール・料金レイヤーと紐付いていない。「Bだと何のモジュールが増えるか」が無い。

### 🟡 D5. クロスセル/アップセル導線が部分的
- SEO(02_03)だけが循環クロスセルを設計済み（優秀）。
- 他モジュール間（相談→実装→制作→SEO→保守）の導線は**実出品では事実上できているのに未文書化**。

### 🟡 D6. 半完成テーブルが多数（体感の「甘さ」の主因）
- CMS運用設計Lv1-3空欄／コーポレート・サービスサイトLvの価格空欄／メディアサイト空欄／カンプなしLv1空欄 等。

## あるべき方向（共通土台）

イラストと同じ骨格に揃える。**Webを"粒度の揃った6工程モジュール"へ再整理**する。

```
[相談/診断]→[戦略・設計]→[デザイン]→[実装/CMS]→[SEO・改善運用]→[保守・環境]
 サポート     IA/訴求/Br   WF→UI    コーディング   サブスク階段     サーバー/WP/ドメイン
   ▲入口        ▲A/Bの分岐         ▲再現式あり    ▲循環クロスセル  ▲周辺収益
   └──── 横：連結＝クロスセル ／ 各モジュール内 縦：Lv＝アップセル ────┘
```
- 各モジュールに **Lvの意味を1つに固定**（難度/作り込み or 関与深度）。
- 見積もり＝**モジュール積み上げ式**に統一（コーディングの式を全体へ展開）。
- **A/B＝乗せるモジュールの差**に再定義（A＝設計軽め＋実装中心／B＝戦略・訴求を厚盛り）。
