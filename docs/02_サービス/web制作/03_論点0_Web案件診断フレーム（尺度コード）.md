# 03. 論点0：Web案件診断フレーム（尺度コード）v0.1

> 2026/6/11 作成（思考実験ドラフト・実案件不要で論理駆動）。
> 目的：Webクライアントの**ふわふわ（事業・マーケ・ブランディングの尺度不足）を座標化**し、
> コーディングサービス並みの「入口ゲート＋再現可能な設計の起点」を上流工程にも与える。
> イラストの**スタイルコードSOP**（感覚→コード化）のWeb版。
> → 物差し：`02_設計診断と改善論点.md` 論点0／`../../01_共通基盤/00_設計思想（共通の根っこ）.md`

## 1. 設計原理

1. **コーディングの強さ＝入口での要件凍結**だった。「カンプ確定（＝デザイン層が凍結済み）」しか
   受けないゲートが、ふわふわを遮断し現場を回した。
2. これを**全工程に一般化**する：各工程モジュールは「**直上の層が一定の成熟度に達していなければ着手しない／
   その層を先に売る**」。
3. よって診断＝**「案件が何か（オブジェクト）」と「クライアントの入力がどこまで固まっているか（成熟度）」を
   分けて測る**こと。後者こそが"尺度合わせ装置"の本体。

---

## 2. コード体系 v0.1

コードは2部構成：**`[オブジェクト] // [成熟度ベクトル]`**

### パート1：オブジェクト（何を作るか）

| 軸 | ラベル | 補足（難度の向き） |
| --- | --- | --- |
| **G 目的**（複数可） | G1認知/ブランディング・G2集客・G3販売/CV・G4採用・G5信頼/権威 | 目的が多いほど難度↑ |
| **F 形態** | F1 LP・F2 コーポレート/サービス・F3 EC・F4 メディア・F5 Webアプリ/動的 | 既存「技術難度モデル」と対応 |
| **Sc 規模** | Sc1 単一・Sc2 小階層・Sc3 中階層・Sc4 大規模 | ページ数・階層 |

### パート2：成熟度ベクトル（入力がどこまで固まっているか）★装置の本体

**上流4層**を、共通の成熟度スケールで評価する。

共通スケール：
| ラベル | 意味 |
| --- | --- |
| **M0 無** | 丸投げ・未着手（「いい感じに」） |
| **M1 断片** | 感覚・口頭・参考URLのみ。言語化されていない |
| **M2 言語化** | ドキュメントとして言語化済み |
| **M3 確定資産** | 確定し、成果物/データとして存在（例：Ds=M3＝カンプ確定） |

| 層 | コード | 中身（クライアントが固めるべきもの） |
| --- | --- | --- |
| 事業・市場 | **Bz** | 誰に/何を/なぜ売れるか（3C・KGI・収益構造） |
| ブランド | **Br** | ↓サブ軸で分解 |
| 情報設計 | **IA** | サイト構成・導線・各ページ構成 |
| デザイン | **Ds** | ワイヤー→デザインカンプ |

> 下流の **Cd 実装 / Op 運用・SEO / Mt 保守** は「成熟度」ではなく**工数**で測る領域（既存の
> コーディング積み上げ式・SEOレベル階段が担当）。成熟度ベクトルは上流4層に限定する。

### ブランド層(Br)のサブ分解（スタイルコード的に）

ブランドこそ尺度不足の温床なので、5サブ軸に割って各M0〜M3：
| サブ軸 | 内容 |
| --- | --- |
| Br-P | パーパス/コンセプト（理念・存在理由） |
| Br-V | 提供価値/USP（差別化・売り軸） |
| Br-T | ターゲット/ペルソナ |
| Br-M | トーン&マナー（言語・世界観） |
| Br-I | VI（ロゴ・カラー・フォント） |

### コード表記例

```
案件A（ふわふわMAX）： G2,3 / F2 / Sc2 // Bz:M1 Br:M0 IA:M1 Ds:M0
 → 集客+販売のコーポレート中規模。だが戦略も設計もデザインも未成熟、ブランドは丸投げ。

案件B（コーディング案件）： G3 / F1 / Sc1 // Bz:- Br:- IA:M3 Ds:M3
 → デザイン確定＝既存の入口ゲートを通過済み。即実装できる状態。
```

## 3. 判定ロジック（どこから売るか）

### ゲート原則（コーディングの一般化）
> **各工程モジュールは、直上の層が M2 以上（デザイン実装はDs=M3）でなければ着手しない。
> 満たさなければ、その未成熟層を"先に売る"。**

### エントリー判定
- 成熟度ベクトルを**上流(Bz)から順に見て、最初に M<2 になる層**＝「いま売るべきモジュール（エントリー）」。
- そこを M2/M3 へ引き上げ→下流へ進む。これが**クロスセルの自然な順路**。
- 全層M3なら即実装＝コーディング案件。

### 全体の「尺度レベル」＝思考難度Lvに接続
| エントリー層 | 思考難度Lv（既存マクロモデル） | 意味 |
| --- | --- | --- |
| Ds | Lv1〜2 模倣再現/要素選択 | カンプ寸前。手を動かす段階 |
| IA | Lv3 構成設計 | 構成を組み立てる |
| Br | Lv4 概念抽出 | 抽象（理念・世界観）を言語化 |
| Bz | Lv5 文脈創出 | 何を売るかから設計 |

→ **エントリーが上流なほど難度・単価の天井が上がる**（価格設計＝論点3に接続）。
→ ふわふわ案件＝エントリーが上流（Bz/Br）。そこを**「診断/設計」モジュールとして正当に売る根拠**になる。

- 受付ヒアリングで**このコードを埋める**＝花田の人力診断をフォーマット化（＝「相談」の正体を商品化）。
- **コード確定＝認識合わせ完了**（イラストのスタイルコード完了条件と同じ思想）。
- 確定コードから**エントリーモジュールを提案**：ふわふわなら「まず診断・設計を売る」が論理的に正当化される
  （値引き交渉でなく、工程として必要だから）。
- このロジックは**将来CRMの受付・見積もりロジックにそのまま実装**できる（→ `../../03_将来構想/自社CRMシステム構想.md`）。
