# 04. 論点0-2：変換ロジック（ふわふわ→構造化）v0.1

> 2026/6/11 思考実験ドラフト。論点0（尺度コード `03_…`）の運用の心臓部。
> 問い：コーディングは「カンプ無し＝門前払い」で回せたが、上流のふわふわ客を門前払いすると
> **市場ごと消える**。では、どう"門前払いせずに構造化された案件へ変換"して収益化するか。

---

## 1. コアの発想：二択に CONVERT を挿す

コーディングの応答は2モードしかなかった：
```
EXECUTE（カンプ確定＝実行）  /  DECLINE（カンプ無し＝出直し）
```
Web全体では、その**間に CONVERT（変換）**を挿す。これが最大の違いであり収益源。

| モード | 条件 | アクション |
| --- | --- | --- |
| **EXECUTE 実行** | 必要層がM3まで揃う（凍結済み入力あり） | そのまま実行モジュール（コーディング等）。＝既存ゲート通過 |
| **CONVERT 変換** | 必要層が未成熟（M0〜M1） | **門前払いしない**。その層を上げる「構造化モジュール」を先に売る ★主戦場 |
| **DECLINE/DEFER 辞退** | 構造化にも対価を払う気がない（予算ゼロ／丸投げ／相見積もり狙い） | 門前払い。**ここだけは断る** |

> 旧モデルは「実行 or 門前払い」しか無く、ふわふわ客は**花田が無償の人力で握り込む**しかなかった。
> CONVERTを商品化することで、握り込みを**有償の工程**に変える＝属人性を排しつつ主導権は維持。

## 1.5 二軸モデル：誰を（サクセスレベル）× どこから（成熟度）★v0.1修正

当初「診断は有償か撒き餌か」「設計を先に買わせる見せ方」を急所と置いたが、**両方とも既存の
「クライアントサクセスレベル」足切りに吸収される**（2026/6/11 議論で確定）。本ロジックは2軸で動く。

- **軸1：クライアントサクセスレベル（WHO・誰を）** ＝ 受けるか/育てるか/距離を取るか
  （出典：Notion「05_クライアントサクセスレベル対応マニュアル」。**Win-Win or No Deal**）
- **軸2：成熟度ベクトル（WHAT-state・どこから）** ＝ どのモジュールから入るか（`03_…`）

### クライアントサクセスレベル（Lv1-5・要点）
| Lv | 関係性 | 方針 |
| --- | --- | --- |
| Lv5 共創型 | win-win-win | 共同戦略・継続・成果/定額モデル |
| Lv4 安定型 | win-win | テンプレ化し継続契約へ |
| Lv3 不安定型（教育案件） | win-win→lose | 境界明示＋見える化で**育てる**（CONVERTの主戦場） |
| Lv2 摩耗型 | win-lose | 線引き徹底→有料相談に誘導。越えたら見送り |
| Lv1 搾取型 | lose-lose | 即距離／丁寧に辞退 |
> 評価軸：技術理解度・情報提供精度・予算/納期条件・コミュ円滑さ（＝ビジネススキル）。

### 二軸の決定マトリクス
```
                成熟度：高(Ds/IAがM2-3)        成熟度：低(Bz/BrがM0-1＝ふわふわ)
サクセス Lv4-5   EXECUTE（即実行）             CONVERT（共創的に上流から一緒に作る）
サクセス Lv3     EXECUTE / 軽CONVERT          CONVERT（有償で構造化＝教育案件の主戦場）
サクセス Lv1-2   ―（基本DECLINE。診断課金で自然脱落）―
```
- **サクセスLv≤2 → DECLINE/選別**が先（成熟度を見るまでもない）。「サクセスレベル2以下は付き合いが渋い」。
- **サクセスLv≥3 → 受ける**。その上で成熟度ベクトルが EXECUTE か CONVERT かを決める。
- ふわふわ（成熟度低）でも**サクセスLv3以上なら門前払いせずCONVERT**＝これがWeb版の主戦場。

> 重要：**有償診断は「サクセスレベルを測る装置」を兼ねる**。診断に対価を払う意思＝予算条件＋
> 協業姿勢のシグナル。Lv1-2（搾取型・予算先行型）は診断課金の時点で自然に脱落する＝**足切りの自動化**。
> ゆえに「診断を撒き餌（無償）にする」は、拾ってはいけない層を拾う悪手。**有償が正しい**。

### Lv1-2の精緻化（バックテスト ケース3 反映）
一律DECLINEにせず、**有償診断を踏み絵**にする：
- **Lv2** → 有償診断に乗せる。買えば受ける（**防御付き**＝着手金・線引き・有料ディレクション）。
  拒否・ゴネなら DECLINE。
- **Lv1** → DECLINE（搾取型）。

### サクセスレベルは2段判定（バックテスト ケース3 反映）
- **一次判定（無償）**：初期問い合わせの言い回しで粗く当てる（Notion「05_…」見抜きポイント表）。
- **確定判定（有償診断内）**：診断で成熟度コード化と同時にLv確定。
- Lv1-2濃厚なら**診断前に線引き**（撒き餌診断はしない）。

## 2. 変換先：各上流層に対応する「構造化モジュール」

未成熟な層を、**その層を凍結する成果物を作る有償モジュール**に変換する。
各モジュールの**成果物＝次工程の「M3資産（凍結済み入力）」**になるのがミソ（連鎖する）。

> ★ モジュールの定義・成果物・**価格は `06_論点3…` の「上流モジュール価格マスタ」が単一の正本**。
> ここでは変換ロジック上の役割だけ示す（価格はマスタ参照、二重管理しない）。

| 未成熟な層 | 変換先モジュール | 成果物（＝次のM3資産） |
| --- | --- | --- |
| Bz 事業・市場 | 事業・市場ディスカバリー | 事業理解ドキュメント |
| Br ブランド | ブランド設計 | ブランドガイドライン（P/V/T/M/I） |
| IA 情報設計 | 情報設計(IA) | IA設計書（サイトマップ・導線） |
| Ds デザイン | デザイン（カンプ制作） | デザインカンプ（＝コーディングのゲートに合流） |

→ これは**クロスセルの自然な順路そのもの**（論点5へ直結）：
`診断 → Bz → Br → IA → Ds → 実装(Cd) → 運用(Op) → 保守(Mt)`
→ ①設計・ブランディングフェーズの総額＝**エントリー層から下のこれらを各作り込みLvで積んだ合計**（`06_…` ①）。

## 3. 変換の着火点：「Web診断」＝尺度コードを確定する有償の最小商品

旧「相談 ¥3,000（人力握り込み・実質フリーコンサル）」を、**構造化された有償診断**へ置き換える。

- **中身**：ヒアリング → 尺度コード確定（`03_…`）→ 推奨ロードマップ（どの層から着手・概算レンジ）。
- **成果物**：①案件の尺度コード ②着手順ロードマップ ③概算。クライアントが持ち帰れる価値物。
- **役割**：CONVERTの着火点。**構造を見せることで「上流から着手すべき」が論理的に腑に落ちる**
  （値引き交渉でなく、工程として必要だから、という納得）。
- **アップセル動線**：上流モジュール契約時は**診断費を充当（クレジット）**＝次工程への自然な接続。

## 4. フリーコンサル防止ルール（旧モデルの失敗を制度で回避）

旧モデルの本質的弱点＝**尺度合わせを無償の人力で抱えていた**こと。これを境界線で防ぐ。

- **境界線**：「**現状把握（無償）**」と「**構造化（有償）**」の間に線を引く。
  - 無償でやってよい：見積もり提示に必要な最小ヒアリング（コードの**オブジェクト部 G/F/Sc** の確認まで）。
  - ここから先（**成熟度を引き上げる作業＝Bz/Br/IAの言語化**）に入った瞬間に**有償（診断 or 構造化モジュール）**。
- ゆえに「とりあえず方針だけ相談したい」は**Web診断（有償）**に乗せる。無償ディスカッションで設計を渡さない。

## 5. CONVERT判定のしきい値（どこで変換を挿すか）

```
依頼された下流モジュールの「直上の必要層」を見る
 ├─ M3            → EXECUTE（そのまま実行）
 ├─ M2            → 原則EXECUTE（軽い確認付き。Dsだけは M3 必須）
 ├─ M0〜M1        → CONVERT（その層の構造化モジュールを先に売る）
 │                   └─ 未成熟層が複数 → 最上流から順に積む（Bz→Br→IA→Ds＝ロードマップ）
 └─ 対価意思なし  → DECLINE
```

- **「凍結なき実行」は受けない**：客が「実行だけ安く」と上流すっ飛ばしを要求しても、凍結入力が無ければ
  EXECUTE不可。リスクを明示し、CONVERT（上流を先に）か DECLINE に倒す。
  ＝コーディングのゲート思想を全工程で維持。
- **ゲート達成は2ルート**（バックテスト ケース4 反映）：ゲートの条件は「直上の層がM3(デザイン)/
  M2(その他)に達すること」**だけ**。達成手段は ①**自社CONVERTを売る** ／ ②**クライアントが
  自前・外部で凍結資産を用意する** の2つ。コーディングのカンプ支給は②。
  **CONVERTは提案であり強制ではない**（客が②で揃えるならEXECUTEに合流）。

## 6. リスクと価格の接続（論点3への布石）

- **ふわふわ（上流M低）ほど手戻りリスクが高い** → それが「上流を先に固める」必要性の根拠であり、
  CONVERTモジュールの対価の正当化。
- 上流を飛ばして実行する場合は、**リスクプレミアム or 着手金**を乗せる（凍結が無い分の保険）。
- 単価の天井：エントリー層が上流なほど高い（Bz=Lv5 … Ds=Lv1-2、`03_…` の対応表）。

---

## 7. 漏れ（リーク）対策

- 診断/構造化の成果物（ロードマップ・設計書）を**他社実装に流用される**リスク（イラストのラフ流用と同型）。
- 対策：①成果物の利用範囲を明示（自社検討用）②**診断費・設計費を本契約に充当**して囲い込む
  ③上流成果物は「実装まで含めて価値が出る」設計にし、単体流出の旨味を下げる。

---

## 8. この設計が解くもの

- 成因の核心（尺度合わせ＝花田の人力＝属人化）を、**CONVERT＝有償の構造化モジュール**に外在化。
- ふわふわ客を**門前払いせず収益化**（市場を捨てない）。
- 結果として**クロスセルの連鎖**が自然に立ち上がる（論点5）／**CRMの受付・提案ロジック**に実装可能。
